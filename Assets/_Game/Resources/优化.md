# 项目健康度与扩展性全量诊断

> 审查范围：`Assets/_Game/Script` 全目录
> 初次审查：2026-04-23 · 复审更新：2026-04-23

---

## 一、架构级问题

### 1. EliminateCardController 上帝对象 — 2690 行

纯 C# 控制器承担了消除逻辑、卡牌状态机、Combo、道具效果、复活恢复、分数结算等至少 6 个职责。没有 partial class 拆分，单文件 87KB。

**建议**：拆为 `EliminateCardController.Core.cs`、`EliminateCardController.Combo.cs`、`EliminateCardController.Prop.cs`、`EliminateCardController.Resurgence.cs`。

### 2. EliminateTheAreaEntityLogic — 1927 行

实体逻辑脚本同时管理等待区渲染、分数精灵、卡牌拖拽、特效触发。职责过重。

**建议**：按功能域拆 partial class。

### 3. GameAssetModule — 1764 行

资源加载模块没有 partial class，资源缓存、预制体管理、Sprite 加载全部混在一起。

**建议**：拆为 `GameAssetModule.Cache.cs`、`GameAssetModule.Prefab.cs`、`GameAssetModule.Sprite.cs`。

### 4. PlayfieldEntityModule — 1506 行

场地实体管理模块，宠物站位、投影同步、标记点快照全部在一个文件。

**建议**：拆为 `PlayfieldEntityModule.Marker.cs`、`PlayfieldEntityModule.Snapshot.cs`、`PlayfieldEntityModule.Placement.cs`。

### 5. GameEntry 静态上帝入口 — 无 DI、无测试性

`GameEntry.CustomComponents.cs`（71 行）中所有模块都是 `static` 属性，业务层到处硬编码 `GameEntry.Fruits`、`GameEntry.DataTables` 等。无法做单元测试替换，无法控制生命周期。

**现状**：框架组件（`GameEntry.BuiltinComponents`）与自定义模块（纯 C# `new` 出来的对象）混在同一入口，初始化顺序靠代码行序隐式保证。

### 6. EliminateCardController.Instance 静态单例

纯 C# 对象用 `static Instance` 暴露，生命周期靠 `RebuildPreview` 赋值 / `Dispose` 清空。任何代码随时可访问，null 时 NRE 无防护。

### 7. 模块初始化模式不一致

`GameEntry.CustomComponents.cs` 中：
- `EntityIdPoolComponent`、`EggHatchComponent`、`PetDiningOrderComponent` → `GetComponent`（MonoBehaviour）
- `GameDataTableModule`、`GameAssetModule`、`PlayerRuntimeModule` 等 → `new`（纯 C#）

两类模块的生命周期管理完全不同：MonoBehaviour 跟随 GameObject，纯 C# 对象没有 `OnDestroy`/`Dispose` 退出路径。

## 二、违反架构标准

### 8. 未使用 Addressables

项目仍使用 GF 的 `ResourceComponent`（底层 `Resources.Load`），3 处代码直接调用 `Resources.Load`：
- `WaitingAreaScoreSpriteRenderer.cs`
- `GameAssetModule.cs`
- `ScoreDigitRenderer.cs`

架构标准要求 Addressables。

### 9. AdvertisementModule 闭包 + 字符串插值产生 GC

回调中使用 `$"[AdvertisementModule] 预加载失败：{err?.errMsg}"` 字符串插值，每次触发都分配。`OnLoad`/`OnError` 注册的 lambda 也是闭包，捕获了实例字段。

## 四、其他问题

### 17. VictoryFailUIForm / DailyChallengeUIForm 直接访问框架内部

```csharp
currentProcedure.procedureOwner.SetData<VarInt32>(...);
currentProcedure.ChangeState<MainProcedure>(currentProcedure.procedureOwner);
```

`procedureOwner` 是 `ProcedureBase` 的内部属性，业务层不应直接访问。2 个 UI 文件存在此问题。

### 18. PropPurchaseUIForm 的 static 事件 + static 状态

`s_purchasedThisBattle` 和 `OnPropPurchased` 都是 static。如果异常退出未走 `ResetBattlePurchaseState`，下一局会继承脏状态。static 事件也容易因订阅方未反订阅而泄漏。
